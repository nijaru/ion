# Ion Roadmap: Stable Native Agent First

## Answer First

Ion should now move in four deliberate phases:

1. **Finish the minimal native coding agent.** Keep the runtime Pi-simple: prompt, stream, tools, compact, cancel, retry, persist, resume, continue.
2. **Polish the core TUI/CLI shell.** Make the first screen, transcript, queue, slash commands, sessions, and provider/model UX feel closer to Claude Code/Droid while preserving Pi's simplicity.
3. **Restore safety and product table stakes.** Reintroduce trust, modes, approvals, sandbox posture, and session controls only after they cannot corrupt or obscure the core loop.
4. **Add advanced agent features from evidence.** ACP/subscription bridges, subagents, skills, routing, privacy, optimizer loops, and branch/tree workflows wait until the solo native loop is stable and instrumented.

Ion is the product. Canto is the mechanism. Ion should not absorb Canto, and Canto should not ship Ion's product policy.

## Roadmap

| Phase | Name | Goal | Work |
| --- | --- | --- | --- |
| I0 | Stabilization closeout | Prove the current native loop is boring under deterministic, race, tmux, and live-provider checks. | Keep `tk-mmcs` active; close only after fresh turn, tool turn, compact, cancel, provider error, resume, and follow-up are validated. |
| I1 | Pi-level core parity | Keep Ion usable as a minimal coding agent with clean scripting. | `-p`, stdin, JSON, explicit resume/continue, compact, file/bash tools, transcript durability, no duplicate/provider-invalid history. |
| I2 | Minimal polished TUI | Make daily terminal use readable and predictable. | Transcript spacing/markdown, compact tool display with expand path, slash autocomplete, `/session` and `/resume` polish, queued input visibility/recall, settings clarity. |
| I3 | Safety/product table stakes | Re-enable safety and permissions without fighting the loop. | READ/EDIT/AUTO semantics, workspace trust, approval UX, sandbox state display, stale error clearing, provider/model/favorite state hygiene. |
| I4 | Advanced surfaces | Add features only when backed by stable traces/evals. | ACP/subscription backends, ChatGPT subscription path, subagents, skills, MCP expansion, routing/cascades, privacy redaction, branch/tree workflows. |
| I5 | SOTA/eval-driven optimization | Use Ion traces to improve prompts/tools/configs offline. | DSPy/GEPA-style eval trace artifacts, prompt/tool-description candidates, model-routing experiments, no self-mutating runtime prompts. |

## Immediate Next Work

1. Keep `tk-mmcs` as the active Ion umbrella.
2. Finish open P1/P2 table-stakes defects before reopening advanced tasks:
   - runtime config / active preset mismatch from CLI overrides
   - transcript/TUI readability and command autocomplete regressions
   - session browser and resume targeting polish
3. Keep Canto core-contract audit closed unless an Ion smoke exposes a concrete Canto-owned defect.
4. Run verification in this order for native-loop changes:
   - focused package tests
   - `go test ./... -count=1`
   - native race subset
   - tmux text capture for TUI changes
   - Fedora local-api live smoke when available, then free/cheap OpenRouter fallback

## Non-Goals Right Now

- No broad Canto redesign from Ion.
- No ACP/subagent/privacy/routing work during I0-I2 unless it blocks the solo native loop.
- No provider/framework feature chasing without a failing Ion contract or an explicit roadmap promotion.
- No runtime DSPy/GEPA prompt mutation.
- No external filesystem dependency before Ion's local workspace model is stable.
